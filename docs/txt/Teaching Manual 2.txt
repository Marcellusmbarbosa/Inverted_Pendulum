The Getting Started manual guides you through the real-time control of pendulum-cart
system, shown in the demo form.  The External Interface manual shows how to build a
controller of your own design in the C language. Table 1 describes the content of each
Teaching Manuals (TM). Any controller comprises two essential components: a
mechanism for knowledge representation and a mechanism for decision-making. With
conventional controllers a mathematical model (such as a set of differential equations or
transfer function) represents knowledge and decision-making based on optimisation.
When the system is complex or less precisely known it becomes difficult to characterise
the knowledge adequately with conventional means. One must find alternative ways to
encode the available knowledge about the system. It can result in the development of
intelligent controllers. To this group, among others, belong: rule-based and fuzzy
controllers.
Before exploring the examples you need to know if the system model is compatible with
reality. The default parameters are stored in the model but it cannot be guaranteed that
they are well identified. It is recommended to go through the identification procedure and
experiments described in Teaching Manual 1 “Modelling, Identification, PID and LQ
Controllers”.
The position measurement signals from the encoders play a crucial role in achieving the
control goal. Measurement technique is another factor influencing the quality of control.
The velocities in the system are not measured, they are observed. A poor observation
algorithm may be a source of serious errors.
Remark. In many cases users would like not only to modify parameters of rule-based
controllers embedded in the Real Time Kernel but also to develop their own algorithms of
control. Therefore we shall try to clearly explain how to do it. The methodology used for
LQ, PID or Fuzzy controllers (see the next section) can not be used directly for Rule-
based controllers. For LQ, PID or Fuzzy techniques we are able to give a ready controller
structure written in C. It does not matter if the controller algorithm is prepared as a fixed
DLL  or an external DLL file. The  user can take advantage of the flexibility of the structure
to create his own algorithms and is only limited to a number of input/output signals
defined in a given LQ, PID or Fuzzy structure.
The rule-based toolbox structure offers also certain flexibility allowing to modify control
algorithms (limited to parameter change). However, it might not be sufficient for more
creative users.

There are two ways to build a rule-based controller significantly different from the toolbox
rule-based controller:
• implement it in C using the External Interface method,
• implement it in MATLAB/Simulink using RTW (Real-time Workshop from

After these remarks we are ready to describe the model of the toolbox rule-based
controller and then its correspondence to the real system. At the beginning of this manual
you find the model of the cart-pendulum system. It contains several dedicated blocks
added to obtain the best compatibility with the real laboratory system.
This model was presented in Teaching Manual 1 and it is invoked by writing: model in the
MATLAB Command Window. In the Rule-based Controllers section a number of controller
models are given.
These controller models are built to be compatible with the real system controllers.
Sampling and quantization effects are also modelled to correspond to sampling
introduced by the computer into the control algorithm and a finite accuracy of sensors
(encoders) with discrete signals.
The goal of modelling is to achieve time responses of the model and the real system
close to each other, as far as the shapes and amplitudes of signals are concerned. Only
then can we trust our modelling of rule-based controllers. Usually, to build a controller one
starts from simulation models. Then the controllers are implemented in the real system.
Finally the simulation models are verified and modified to match reality as accurately as
possible.
The fuzzy controllers in section 3 are presented in the simulation and real environments.
Three controller demos are given. The Crane Fuzzy Controller is the first and simplest
example. It contains only seven rules related to two input variables and one control
output. It has been built to show the fuzzy technique rather than to develop the most
accurate crane control policy. The second demo – Stabilising Fuzzy Controller emphasis
a difference between linear  (rule-based) and non-linear (fuzzy) stabilisation policies.
The last demo – Swing-up Fuzzy Controller encourages the user to design more complex
controllers using the fuzzy technique.
The user is equipped with simulation tools to build his own fuzzy controller. Then he can
try to operate the simulated fuzzy controller with the real system. One can access
pd_fuzzy.m from the MATLAB Command Window, where the external interface file for
fuzzy control is stored. Before starting the practical it is only necessary to load the
generated fismatrix into the fismat fuzzy controller block.

Rule-Based Control
The material presented in this chapter consists of two parts 2.1 and 2.2. Part 2.1
corresponds to simulation of rule-based control. Part 2.2 refers to practicals.
The pendulum-cart system is an interesting laboratory model that can be controlled in
many ways. Four state variables and only one control characterise this non-linear system.
The first variable, cart position, is constrained by the rail length. The control, force
pushing the cart back and forth on the rail, is also constrained.
One can ask if it is possible that a human being can serve as a controller for the system?
The answer is positive if the control algorithm is not too complex. Even such a control task
as pole balancing by hand in its unstable upright position can be satisfactorily done by a
human being. Unfortunately we are not able to balance the pendulum by pushing back
and forth the cart. We fail because we do not respond quickly enough to keep the
pendulum in the unstable equilibrium point. However our experiments are not useless.
We can gather knowledge about a possible control algorithm, we can also build a model
of the controller and simulate its behaviour. We can study time and frequency responses
of the system, generate additional characteristics such as potential or kinetics energy of
the system etc. This information and our reasoning may result in a number of rules finally
put together in the form of a control algorithm.
The rule-based control does not correspond to any of quality criteria such as as time-
optimal, linear quadratic etc. It is rather a policy that achieves the control goal and is
robust and insensitive to parameter changes and disturbances. With the pendulum-cart
system when the control goal is defined as “swing-up and upright stabilisation” we deal
with a case when a trade off between two control actions is concerned. Namely, centring
the cart and swinging-up the pendulum contradicts each other. To swing-up the pendulum
we must put the cart in motion. Control rules applied to the cart centring may contradict
raising-up the pendulum. Despite such effects the control goal has to be achieved finally.
We cannot guess control rules they are obtained by a reasoning process. Therefore the
rule-based approach is classified as an intelligent method. When conventional control
algorithms fail intelligent methods seem to be more promising in solving complex control
problems.

The computer sampling period has also a significant influence on the measured and
control signals. The sampling period is fixed to 0.01 s. There are several reasons to
choose this value. The sampling period 0.01 s means the control frequency 100 Hz. It is
sufficient for the system with the natural frequency of about 1 Hz. Further shortening of
the sampling period is not recommended. It must be a trade off between sampling and
quantization. Quantization is defined by a construction of sensing devices: the encoders.

It is worth noting that too short a sampling period is a source of difficulties in co-operation
between the Real-Time Kernel (RTK) under MS WINDOWS and the visualisation
processes and also leaves too little time for complex control algorithms.

To achieve a soft landing in the vicinity of the upright position, the algorithm checks
whether the kinetic energy of the pendulum minus energy loss due to friction forces is
sufficient to rise up the centre of gravity of the pendulum to its upright position. Look at
condition no.5 in the Soft landing arbiter block (Figure 9). If the condition is satisfied then
the control is set to zero (no.6).
The swinging rule has a very simple form (no. 8). It is “bang-bang” with switching
moments related to the sign of the angular velocity and angle position, which points up, or
down. The cart centring action is based on a heuristic rule. The magnitude of the control
force is reduced only if the actual force would make the cart to run away too far from the
centre of the rail (no.7).

The magnitude is reduced proportionally to the cart position x1 with the coefficient α and
to the quadrate of the cart velocity x3 with the coefficient β (no.11).
How to obtain the best pair (α, β) is an open question. There is a trade-off between the
centring action and swinging up the pendulum. Observations of the system behaviour
encourage us to introduce a third centring possibility. Pushing the cart back and forth has
little effect on the pendulum when the pendulum is in the horizontal position (x2 equals to
π/2). We can assume a small angle γ in the neighbourhood of ±π/2 and apply the centring
action if necessary (no.9 and no.10). We can decide also not to apply any of the centring
rules setting the parameters: α, β, γ equal to zero. In this case the algorithm swings up the
pendulum, increasing its total energy, with no regard to the cart position (no.8). Then the
control is pure “bang-bang”. The Swinging & centring controller model is shown in Figure
10.

There are also a number of effects difficult for modelling. For example, the effects related
to Coulomb friction of the cart. We can compensate these effects by control. If control is
positive (no.12) we can make it “more positive” by adding an amount of control to
overcome the Coulomb friction (no.14). Similarly, we make a negative control “more
negative” (no.13). The Coulomb Friction & Swinging Control Gain block perform this
action (Figure 11). Then the control is normalised to stay in the range [-1, +1] (no.15 and
no.16). Usually the normalisation performed by the Saturation block is not active because
the maximum control value is reduced by the Umax parameter. However usually we do
not use the maximum force M generated by the DC motor. Thus in our disposal we have
the parameter Umax to reduce this force proportionally, ( 0 ≤  Umax  ≤ 1 ). This control signal
is sampled and multiplied by M inside the Dynamics block. The real maximum control
magnitude M is equal to 16.5 N.
In all examples and experiments we assume Umax=0.25 so we use only 25% of the DC
motor power, in fact  35% as the normalised u value is enlarged to 0.35 (0.25+0.1) due to
the Coulomb friction effect, modelled by the Friction parameter (Friction=0.1).
Figure 11: Coulomb Friction & Swinging Control Gain controller
The similar block to the one given in Figure 11, but with different parameters is applied at
the output of the linear controller. Its name is Coulomb Friction & Linear Control Gain. In
the linear case we assume: Umax=1 and Friction=0.1. It means that we use the maximum
force (Umax=1) which can be generated by the DC motor to stabilise the system when the
upright stabilisation zone is achieved. The Friction value is also a significant parameter for
the control goal. We have oscillations of amplitude ±0.1 m of the cart position around the
desired cart position value for Friction=0.1 and we observe dumped oscillations for
Friction>0.1, for example Friction=0.15. These observations are valid for both the
simulated and real-time responses of the system.
Finally, the Limit Guard block (Figure 12) is added to reinforce centring of the cart and
prevent it from overrunning the ends of the rail. The rule is very simple. If the range given
by the Rail mg "rail limit" parameter in the limiter block is exceeded (no.17), then the
opposite to the direction of motion maximum steering force (i.e. umax2=1) is applied to
the cart (no.18).

3 Fuzzy Control
We assume that the reader has studied Section 2. It is advisable because throughout the
present section we make comparisons and refer to certain details of the previous section.
We also assume that the reader is familiar with the fuzzy set terminology. Nevertheless
we give a brief summary of definitions:
• A fuzzy set A, defined over some universe of discourse, is a mapping of that universe
into the interval [0, 1]. The process of creating this mapping is called fuzzification.
• A membership function is the curve that defines how true a given statement is for a
given input value. The membership takes on the value of zero for no membership and
one for full membership. Values in the range 0 to 1 represent partial membership to
the given set.
• The fuzzy set is defined in terms of a pair {x, µ(x)} for all x in the universe of discourse.
The membership function denoted by µ : F→ [0, 1] represents partial or graded
membership to the given set A. For example the fuzzy set representing the term fast is
depicted in Figure 57.

Let A and B be fuzzy sets defined over the universe of discourse X. Their intersection,
A∩B, is defined in terms of  µA(x) and µB(x) as µA∩B (x) = min( µA(x), µB(x) )

Let A and B be fuzzy sets defined over the universe of discourse X. Their union, A∪B,
is defined in terms of  µA(x) and µB(x) as µA∪B (x) = max( µA(x), µB(x) )
• 
Fuzzy reasoning is called the inference mechanism. Fuzzy rules take the form:
If xA is A and / or xB is B ... then yZ is Z
There are fuzzy relations defined over the Cartesian product of the universes of
discourses of the antecedent and consequent variables. For example, the rule R: If x
is A then y is Z, is represented as a fuzzy relation defined as:
µR(x, y) = min(µA(x), µZ(y)).
• 
Interpreting an “If ... then ...” rule involves: evaluating the antecedent (involving
fuzzification) and applying the result to the consequent (implication).
• Defuzzification gives a crisp single value of the possibility distribution. The typical
defuzzification formula has the form:
Mean Of Maximum = argmax(µZ(y))
Note that possibility distribution is different from probability distribution. It need not satisfy
the additivity property of possibility distribution. Possibility distribution places a constraint
on the value that may be taken by the variable in question.
The following steps in the design of a fuzzy controller are essential:
• 
Selection of input/output variables
• 
Scaling the variables (if necessary)
• 
Fuzzification: definition of membership functions for all variables
• 
Fuzzy rules: development of the inference mechanism
• 
Selection of the defuzzification strategy
• 
Re-scaling the variables (if necessary).

The point of fuzzy logic is to map the input space to the output space, and the primary
mechanism for doing this is the inference system which consists of “if  - then” statements
called rules. The “if” part of a rule is called the antecedent, while the “then” part of the rule
is called the consequent. In fuzzy logic if the antecedent is true to some degree, then the
consequent is true to that same degree. More then one rule can be active for the same
input value. It means that membership functions partially overlap one another. A few rules
can interpolate the universe of discourse. The fuzzy control design can be described as
building a control algorithm by capturing the knowledge of operation of the process and
encoding this knowledge in terms of rules.
In this section we discuss how to create and edit fuzzy inference systems by hand using
the MATLAB Fuzzy Logic Toolbox. This toolbox offers two methods of building a fuzzy
controller: interactive using the graphical tool and automatic using clustering and the
adaptive neuro-fuzzy mechanism. There are also two types of fuzzy inference system:
Mamdani and Sugeno. We focus on the Mamdani inference system edited interactively by
hand. This approach makes it possible, in a direct way, to understand consequences of
modifications introduced into the inference mechanism and membership functions.
The MATLAB Fuzzy Logic Toolbox is designed for simulation purposes. Our goal is to use
it also to control the real system. Therefore we have created a toolbox dedicated to the
pendulum-cart system. The name of this toolbox is pd_fuzzy. The main parts of it are
three mechanisms: simulation using the pendulum-cart Simulink model, real-time
experiment using the Fuzzy RTK (Real Time Kernel) and the editor of the fuzzy
pendulum-cart controller using the MATLAB Fuzzy Logic Toolbox.
In the pd_fuzzy toolbox one can also find a number of tools. Some of them are necessary
for a real-time control experiment others are optional and auxiliary.
The mechanism used to create a fuzzy controller can be applied successfully if it meets
the specifications of the pd_fuzzy toolbox. These specifications are reduced to the
number and order of input/output signals. A fuzzy controller must be equipped with four
inputs (the state variables) and one output  (the control). The order of variables is the
following: cart position, pendulum angle, cart velocity and pendulum angular velocity.

section 3.4 we described how to run a controller of your own design.

The fuzzy controller is tracking the square wave signal of the desire cart position. The

then the desired cart position signal will be tracked with a static error. Poor tracking
occurs also for a Friction parameter value much different from the static friction of the real
system.

control shows a high frequency chattering which does not occur in reality.
3.2 STABILISING MODE
3.2.1 FUZZY SET
The stabilising control problem refers to the situation when the pendulum is in its unstable
upright position and the cart is tracking a desired signal (remember to start the system
with the pendulum in its upright position). In the Rule-based controller section a linear
proportional controller is applied as a stabilisation controller. In this section we apply a
fuzzy non-linear controller.
If you are in the MATLAB Command Window then write:
>>fuzzy pd_stab

clicking the ‘Edit pd_stab FIS System’ button in the main pd_fuzzy window . This time all
four state variable inputs are used. We obtain fuzzy variable by constructing the minimum
set of membership functions: two membership functions for each variable. The same
applies also to the output control variable. The eight rules are sufficient to achieve
stabilisation. The stabilisation project is shown in Figure 76.

3.3 SWING-UP MODE
3.3.1 FUZZY SET
In the description of this demo we focused mainly on control aspects of pendulum-cart
system. If reader is interested in editorial options of the MATLAB Fuzzy Toolbox would
refer to sections: 3.1 and  3.2. Next is explained how  we build a swing-up fuzzy controller
equipped with a cart centring policy. In the RTK swing task , the rule-based control
algorithms not concern directly with Swinging & centring controller is left out. You can
easily recognise which part of the old rule-based controller (see Figure 2) has been
replaced by the Fuzzy swing controller (see Figure 102). Ideally we could replace all rule-
based controllers by a number of corresponding fuzzy controllers? For example, combine
the stabilising controller from section 3.2 with a new built swing-up controller and other
rule-based controllers designed in the fuzzy technique? It can be done in simulation.
Unfortunately in the real time there are restrictions given by the structure of RTK. We can
define a number of fis-matrices but we can embed only one fis-matrix in RTK. Being so
restricted we have to fix the number of input and output fuzzy variables and also define
the sequence of consecutive variables. The following input/output structure is determined:
4 inputs and 1 output. The sequence and definition of input variables are: cart position,
pendulum angle, cart velocity, pendulum angular velocity. The definition of output variable
is: force applied to the cart. Of course when building your own controller you do not need
to use all input variables. Membership functions for each variable could be choose
differently by each designer. You can create: single if...then, double if...and....then, triple
if...and...and...then or quadruple if...and...and...and...then rules. You can use also the “or”
logic operator instead of the “and” operator. This is the flexibility given by the fuzzy -logic.
Another question is: can one build a general fuzzy controller having the above described
structure and replacing the entire rule-based functionality? The answer seems to be
positive. Nevertheless, the problem is quite complex. We leave it as an open question for
a creative user.

3.4 HOW TO BUILD AND RUN YOUR OWN FUZZY CONTROLLER
Follow the steps for simulation:
1. 
Define your own fuzzy controller. Remember that the fuzzy controller structure
is specific and fixed. The structure of RTK impose restrictions. In principle you
have two ready structures: swing-up RTK, and crane or stabilising RTK. The
Alpha offset parameter of the Real Time Task block make it possible to
distinguish the crane RTK from the stabilising RTK. In the case of the
stabilising RTK the Alpha offset parameter is set to 0. In the case of the crane
RTK this parameter is set to π.
If you decide to build a general fuzzy controller (excluding all rule-based
algorithms) then the swing-up RTK structure must be used.
2. 
Build a reference model of the fuzzy controller using FIS Editor from MATLAB
Fuzzy Toolbox.
3. 
Save your xxxx.fis file to the directory containing other *.fis files i.e.
pd_crane.fis, etc.
4. 
Build a Simulink controller. For example modify the pd_mswing.mdl model
(see Figure 102)
5. 
Write
>>fismat=readfis(‘xxxx’)
in the MATLAB Command Window. The “fismat” variable is created and  ‘xxxx’
Fismatrix is loaded under this name to the MATLAB Workspace.
6. 
Start simulation.
And for Real-time :
7. 
To perform the practical you must refer to the control problem definition. For
example if your controller performs the stabilisation task then refer to the Real
Time Task mask given in Figure 86. Change the name in the FIS system name
window: put xxxx.fis instead of pd_stab.fis.
8. 
Start practical
